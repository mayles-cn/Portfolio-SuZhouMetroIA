# 严格 PM 质量门禁

本文档用于约束版本迭代时的体验、逻辑与工程质量底线。

## A. 用户体验门禁（必须通过）

1. 全局 UI 文案必须为中文，不允许乱码。
2. 助手回复必须可见流式输出，禁止一次性整段输出。
3. 一次语音按下到释放，只生成一条用户语音消息。
4. 路线规划图不得出现重叠严重、不可辨识的绘制结果。
5. 结算面板信息完整：票种、单价、数量、实付、出票结果。

## B. 业务逻辑门禁（必须通过）

1. 站点未命中时，必须返回候选站点。
2. 票价估算失败必须返回结构化错误，不可无提示。
3. 地图“去这里”必须能进入结算流程。
4. 快速购票与路线购票必须正确预填结算数据。
5. 支付成功后必须触发聊天区初始化（非空白清空）。

## C. 稳定性门禁（必须通过）

1. ASR 通道断开后可自动恢复。
2. LLM 请求超时可自动重试一次。
3. 同时仅允许一个模型请求在途，排队仅保留最新请求。
4. 工具调用失败不应导致 UI 卡死或主流程中断。

## D. 样式与配置门禁（本次新增）

1. 新增/修改控件样式优先写入 `config/ui_styles.json`。
2. 控件调用样式统一使用 `UiStyleService::style(key, fallback)`。
3. 禁止在业务代码中新增无 key 管理的硬编码 QSS。
4. 样式 key 命名应遵循：`模块.控件.状态`。
5. `ui_styles.json` 缺项时，fallback 样式必须保证可读可用。

## E. 日志与可观测性门禁

1. 每轮模型回复需有完整日志链：
   - `llm.request`
   - `llm.http`
   - `llm.response`
   - `llm.stream` / `llm.stream.delta`
2. 工具调用必须记录调用与结果。
3. 错误日志中不能泄露密钥内容。

## F. 发版前检查清单

1. 编译检查：关键变更文件必须可独立编译。
2. 交互检查：F7/F8/F9 快捷键功能正常。
3. 语音检查：单次语音输入仅提交一次。
4. 购票检查：快速购票与路线购票均可出票。
5. 样式检查：`ui_styles.json` key 与代码引用一致。
6. 文档检查：README 与 docs 与代码行为一致。
